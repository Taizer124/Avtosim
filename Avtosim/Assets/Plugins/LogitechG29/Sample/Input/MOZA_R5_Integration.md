# Интеграция руля MOZA R5 (ввод)

> Статус: **РУЛЬ + ПЕДАЛИ (ГАЗ/ТОРМОЗ) ПОДКЛЮЧЕНЫ, КНОПКИ — НЕТ**.
> Steering, Throttle и Brake уже вшиты в `InputController.inputactions` и
> работают наравне с Logitech/клавиатурой. Педали подключены **по данным
> открытого проекта AZOM, не проверено на реальном железе** — см. раздел
> "Педали" ниже. Кнопки на руле по-прежнему ждут данных (см. ниже, почему).

## Откуда взяты данные

Разобрал бинарный файл `F:\MOZA R5 Base.inputtrace` (записан через
Window → Analysis → Input Debugger → Record в Unity). Это подтверждённые
факты, не предположения:

- Устройство: `HID::Gudsen MOZA R5 Base`, наследуется от layout `Joystick`,
  формат `HID` — то есть MOZA R5 Base определяется как **generic HID-джойстик**,
  никакого проприетарного SDK для чтения ввода не требуется (SDK нужен только
  для Force Feedback, см. ниже).
- Ось руля — control `stick/x` (offset 0, 16 бит, normalize 0..1) —
  **в записи она реально двигалась** (диапазон 57857–58369), значит биндинг
  рабочий.
- Есть вторичная аналоговая ось `rz` (HID usage `"Twist"`) — тоже шевелилась
  в записи (диапазон 31776–31910). Физически не идентифицирована — вероятно
  крутилка/твист-элемент на руле, но без отдельной проверки утверждать не буду.
- Кнопки: раскладка резервирует **128** generic-кнопок (`trigger`,
  `button2`…`button128`, побитово упакованы в байты 17–33 HID-отчёта) плюс
  `hat` (D-pad, 4 бита в байте 17). Это стандартный "на все случаи" HID-фоллбэк
  Unity, а не реальная разметка руля — реальных кнопок на фото гораздо меньше
  (N, WIP, BOX, P, FL, PL, Y, X, B, A, CAM, R, S1, S2, HOME, MENU, RADIO,
  START, D-pad).

## Почему кнопки не добавлены

Записанная трасса — это **0.13 секунды и 50 событий**, за которые вообще ни
одна кнопка не была нажата (0 button-событий в файле). В трассе есть только
лёгкое покручивание руля. То есть физически невозможно понять, какой из 128
generic `buttonN` соответствует, например, кнопке `Y` или `BOX` на фото —
этих данных просто нет в файле. Подставлять номера кнопок наугад я не стал:
неверная подпись на схеме = неверно подключённая функция на реальном руле.

### ВАЖНО: почему в трассе нет кнопок, хотя ты их нажимал

Ты записывал все кнопки по очереди, но в файле их 0. Вероятная причина:
**Input Debugger пишет в кольцевой буфер ограниченного размера** — если
запись продолжалась после того, как все кнопки уже были нажаты (например,
ты ещё покрутил руль перед остановкой/экспортом), старые события кнопок
могли вытесниться новыми, и в файл попал только хвост записи (0.13 сек
покручивания руля, 50 событий, буквально ничего от кнопок). Не баг с моей
стороны и не твоя ошибка — так работает ограниченный буфер трассировки.

**Когда будет доступ к рулю снова** — 2 варианта, оба быстрее повторной
попытки трассировки:

1. **Живой просмотр (2 минуты, надёжнее всего).** Открой Unity →
   **Window → Analysis → Input Debugger**, найди `MOZA R5 Base`, дважды
   кликни. По очереди нажимай кнопки — смотри, какой control меняется
   (`button2`, `button17`, `hat/up` и т.д.). Пришли список вида:
   `N=button5, WIP=button6, BOX=button12, ...` (цифры — только пример
   формата). Заодно так же быстро проверишь Throttle/Brake (см. ниже) —
   какая ось реально газ, какая тормоз.
2. **Новая трасса** — остановить запись СРАЗУ после нажатия последней
   кнопки, не трогая руль/педали после этого (либо увеличить размер буфера
   записи в Input Debugger перед стартом). Пришли файл + порядок нажатий.

Как только пришлёшь список или новую трассу — добавлю привязки в
`InputController.inputactions` и подпишу их на твоей фотографии с реальными
проверенными номерами.

## Что уже сделано (руль + педали газ/тормоз)

В `InputController.inputactions` добавлены привязки MOZA:

```json
// карта "Steering wheel", экшен Steering_Stick
{ "path": "<HID::Gudsen MOZA R5 Base>/stick/x", "groups": "MOZA", "action": "Steering_Stick" }

// карта "Pedals", экшен Throttle
{ "path": "<HID::Gudsen MOZA R5 Base>/z", "groups": "MOZA", "action": "Throttle" }

// карта "Pedals", экшен Brake
{ "path": "<HID::Gudsen MOZA R5 Base>/rz", "groups": "MOZA", "action": "Brake" }
```

Плюс новая control scheme `MOZA` (рядом с `Logitech G29` и `Keyboard`).
`InputControllerReader.cs` и игровой код трогать не пришлось — MOZA работает
через те же `SteeringCallback`/`ThrottleCallback`/`BrakeCallback`, что и
Logitech/клавиатура.

### Откуда взялись Z=газ, Rz=тормоз — и почему это НЕ проверено

В самой трассе педали не нажимались (0.13 сек, только руль), поэтому по
данным файла нельзя было отличить, какая из осей `z/rx/ry/slider/dial` —
газ, а какая тормоз (у всех одинаковые generic-теги, кроме `rz`, у которой
есть HID usage `"Twist"`). Запустил ресёрч по открытым проектам, работающим
с MOZA на уровне HID:

- **[giantorth/AZOM](https://github.com/giantorth/AZOM)** (`Protocol/MozaHidReader.cs`,
  живой поддерживаемый SimHub-плагин) — в коде явно разделены два случая:
  педали через отдельное USB-устройство и **педали через сам хаб** (это
  наш случай — R5 Base + встроенные педали). Для второго случая:
  **`Z` → Throttle, `Rz` → Brake, `Slider` → Clutch (3-я педаль), `Dial` →
  Handbrake**. `Rz` там же прямо назван HID usage "Twist" — это совпадает
  с тем, что я сам нашёл в твоей трассе, независимое подтверждение.
- Уверенность **средняя-высокая** (реальный рабочий код с явными
  комментариями), но версия прошивки MOZA не указана — обновление
  прошивки теоретически может изменить раскладку.

Поэтому подключил только `Throttle=Z` и `Brake=Rz`. **Не подключал:**
- `Slider`→Clutch и `Dial`→Handbrake — у тебя физически нет 3-й педали
  сцепления, а ручник уже подключён с отдельного устройства
  (`<HID::wch.cn Hand Brake>/trigger`), педальный `Dial` не нужен.
- Направление/диапазон осей (0..1 vs -1..1, не инвертированы ли газ/тормоз)
  — генерик-раскладка Unity объявляет `z`/`rz` в диапазоне -1..1 (не 0..1,
  как у Logitech), реальный диапазон хода педали не проверен. **Обязательно
  проверь при первом подключении**: нажми газ в игре — машина должна
  ускоряться, а не тормозить/буксовать назад. Если перепутано местами или
  инвертировано — скажи, поправлю processors в биндинге за секунду.

**Важно:** сейчас у пользователя только 2-педальный сет MOZA — газ и тормоз.
Сцепления и отдельного H-shifter'а физически нет. Поэтому:
- Экшены `Clutch` и `Shifter 1..7` для MOZA пока НЕ добавляются — они
  остаются на Logitech/клавиатуре (F и 1-7) как единственных источниках,
  пока соответствующее железо MOZA не появится.
- Если позже добавится сцепление/шифтер MOZA — пришли новую трассу, довяжу
  так же по образцу Steering/Throttle/Brake.

## Force Feedback (опционально, позже)

FFB на MOZA — только через **MOZA SDK** (https://mozaracing.com/pages/sdk).
Это аналог `LogitechGsdk` для Logitech: отдельная DLL + обёртка. К вводу
отношения не имеет, делается отдельным этапом. Скажи — сделаю обёртку по
образцу `LogitechSDK/LogitechGsdk.cs`.

## Про «отваливающийся» Logitech

Рантайм Logitech в этом проекте не сломан (идентичен рабочему эталону). Если
руль всё же «отваливается» в билде:

1. Известный воркэраунд Input System: в
   `Library/PackageCache/com.unity.inputsystem@*/InputSystem/InputManager.cs`
   в catch-блоке создания устройства поменять `Debug.LogError(...)` →
   `Debug.Log(...)` (см. README плагина). Это снимает падение при
   повторной енумерации HID.
2. Теперь это уже не критично: на те же экшены повешены клавиатура и MOZA,
   поэтому при кратком пропадании одного устройства управление не умирает
   целиком.

Если проблема воспроизводится — пришли точный лог из билда (`Player.log`),
разберём конкретную причину.
